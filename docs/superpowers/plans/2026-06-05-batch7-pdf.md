# Domain Port — Batch 7: PDF generation Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development. Steps use checkbox (`- [ ]`) syntax.

**Goal:** Render invoices and estimates to PDF server-side via **maroto v2** (Go), exposed as download endpoints, with a download button in the UI.

**Architecture:** No DB layer. A new `internal/pdf` package renders an `*repository.Invoice` / `*repository.Estimate` (which already carry business/client snapshots, line items, totals) to PDF bytes using `github.com/johnfercher/maroto/v2`. HTTP handlers stream the bytes as `application/pdf`. The old jsPDF layout is the visual reference; **pixel-perfect parity is NOT required** (spec explicitly flags this) — a clean, correct invoice/estimate PDF (header, parties, line-item table, totals) is the bar.

**Spec:** `docs/superpowers/specs/2026-06-05-domain-port-decomposition-design.md` (PDF risk note). Old reference: `src/lib/utils/pdf.ts` (business header → title+number → parties → details → line-item table → totals → notes).

**Reference templates:** `internal/repository/invoice.go` + `estimate.go` (the domain data), `internal/http/invoices.go` (handler + route registration), `internal/service/invoice.go`.

**Data available on the domain structs (no new queries needed):** Invoice/Estimate have `InvoiceNumber`/`EstimateNumber`, `Date`, `DueDate`/`ValidUntil`, `BusinessSnapshot`/`ClientSnapshot`/`PayerSnapshot` (JSON `{name,email,phone,address,metadata}`), `Subtotal`/`TaxRate`/`TaxAmount`/`Total`, `CurrencyCode`, `Notes`, `Status`, and `LineItems[]` ({Description, Quantity, Rate, Amount}). The PDF renders from the **snapshots** (point-in-time), not live client rows.

- [ ] **Step 3: Implement** `internal/pdf/pdf.go`:
  - A shared `party struct{ Name, Email, Phone, Address string; Metadata map[string]string }` + `parseSnapshot(json string) party` (json.Unmarshal; tolerate empty/`{}`; metadata may be an object or absent).
  - `RenderInvoice(inv *repository.Invoice) ([]byte, error)` and `RenderEstimate(est *repository.Estimate) ([]byte, error)` — both call a shared internal `render(doc docData) ([]byte, error)` where `docData` holds the common fields (title "INVOICE"/"ESTIMATE", number, date, dueLabel+dueValue ("Due"/"Valid Until"), business party, client party, line rows, subtotal/taxRate/taxAmount/total, currency, notes, status).
  - Use maroto v2 builder: `m := maroto.New(cfg)` (A4, mm), add rows: a header row (business name bold + address/email/phone lines), a title/number block (right-aligned title + number + date/due), a "Bill To" block (client party), a line-items **table** (header row Description | Qty | Rate | Amount, then a row per line item — use `text.New` columns in `m.AddRow(...)`), a totals block (Subtotal / Tax (rate%) / Total, right-aligned), and a notes line. Format money with the currency code + 2 decimals (a small `money(v float64, code string) string` helper → e.g. "USD 27.50" or "$27.50" — keep simple). `doc, err := m.Generate(); return doc.GetBytes(), err`.
  - Keep functions <60 lines — split header/parties/table/totals into helpers. Two-plus assertions where sensible (nil-input guard → error).
  - Reference maroto v2 API: `github.com/johnfercher/maroto/v2`, `.../pkg/components/text`, `.../pkg/components/col`, `.../pkg/components/row`, `.../pkg/props`, `.../pkg/config`. (Consult the maroto v2 README for exact constructors; the test just needs valid non-empty PDF bytes.)
