# Tallyo Infrastructure (OpenTofu + Terragrunt)

GCP infrastructure for Tallyo as reusable OpenTofu modules orchestrated by
Terragrunt. Single project / single region today, laid out so adding a region or
project is a directory copy with zero module edits.

**Deployed surface:** one Artifact Registry Docker repo, one shared
`db-f1-micro` zonal Cloud SQL Postgres instance with three databases
(`tallyo_dev`/`tallyo_stg`/`tallyo_prd`, each with its own user+password), three
scale-to-zero gen2 Cloud Run services (`dev`/`stg`/`prd`) running the single
`tallyo` image, per-env Secret Manager secrets (DB password + `ANTHROPIC_API_KEY`),
per-env least-privilege runtime service accounts. No worker, no Cloud Tasks, no
Cloud Scheduler, no public IP / VPC, no account layer.

Cloud Run reaches Cloud SQL over the built-in unix socket
(`--add-cloudsql-instances` via the `cloud_sql_instance` volume), so:

```
DATABASE_URL = postgres://USER:PASSWORD@/DBNAME?host=/cloudsql/PROJECT:REGION:INSTANCE
```

## Layout

```
infra/
  bootstrap/     one-time, LOCAL state — GCS state bucket + core APIs
  modules/       reusable, env-agnostic OpenTofu modules
    project-services/ artifact-registry/ cloud-sql/ database/ secrets/ cloud-run/
  live/
    root.hcl                 GCS backend + google/random provider codegen + common inputs
    _envcommon/              shared per-module input fragments (DRY) with dependency mocks
    tallyo/                  project.hcl  (project_id placeholder)
      australia-southeast1/  region.hcl   (region placeholder)
        artifact-registry/ cloud-sql/        region-shared
        dev/ stg/ prd/  each: database/ secrets/ cloud-run/   per-env leaves
```

`project.hcl` (`project_id = "tallyo"`) and `region.hcl`
(`region = "australia-southeast1"`) are the only place those values live; every
module receives them via the root `inputs` + generated provider. **No module
hard-codes a project id or region.**

## Tooling

- OpenTofu `>= 1.8` (tested with `v1.12.0`)
- Terragrunt `>= 0.67` (tested with `0.99.4` — note its CLI uses
  `terragrunt hcl format` and `terragrunt run --all`, see below)
- `gcloud`

## Credential-free verification

Runs with no cloud credentials. This is the automated gate.

```bash
# 1. tofu fmt across modules + bootstrap
cd infra
tofu fmt -check -recursive modules bootstrap

# 2. tofu validate each module (and bootstrap) with the backend disabled
for d in modules/* bootstrap; do (cd "$d" && tofu init -backend=false >/dev/null && tofu validate); done

# 3. Terragrunt HCL formatting (terragrunt 0.99 syntax)
cd infra/live
terragrunt hcl format --check

# 4. Terragrunt validate the whole region tree using dependency mocks.
#    TG_OFFLINE=1 swaps the GCS backend for a local one so no GCP creds are
#    needed; dependency outputs resolve from each fragment's mock_outputs.
cd infra/live/tallyo/australia-southeast1
TG_OFFLINE=1 terragrunt run --all validate --non-interactive
```

> Terragrunt 0.99 CLI note: older docs use `terragrunt hclfmt --terragrunt-check`
> and `terragrunt run-all validate --terragrunt-non-interactive`. In 0.99 these
> are `terragrunt hcl format --check` and `terragrunt run --all validate
> --non-interactive`.

## One-time bootstrap (human, needs creds + billing)

The GCS state bucket must exist before any `live/` unit can use remote state.

```bash
gcloud auth application-default login
gcloud config set project <PROJECT_ID>

cd infra/bootstrap
cp terraform.tfvars.example terraform.tfvars   # set project_id/region/state_bucket_name
tofu init && tofu apply
```

Set `state_bucket` in `infra/live/root.hcl` to the bucket name you created (GCS
bucket names are globally unique).

## Parameterize project / region

Edit `infra/live/tallyo/project.hcl` and
`infra/live/tallyo/australia-southeast1/region.hcl`. Nothing else changes.

## Secrets (out-of-band)

The DB password is generated by the `secrets` module. The `ANTHROPIC_API_KEY`
secret container is created empty; add a value out-of-band per env:

```bash
gcloud secrets versions add tallyo-<env>-anthropic-api-key --data-file=-
```

## Plan / apply runbook (human, needs creds + billing — incurs cost)

Do NOT run during automated execution. From `infra/live/tallyo/australia-southeast1`:

```bash
terragrunt run --all plan        # review
terragrunt run --all apply       # DAG order, or apply units individually
```

DAG order is: `artifact-registry` + `cloud-sql` → per-env `secrets` → per-env
`database` → per-env `cloud-run`.

**Image dependency:** a `tallyo:latest` image must exist in Artifact Registry
(Plan 2) before any `cloud-run` apply — the `cloud-run` module takes the image
ref as input (`<region>-docker.pkg.dev/<project>/tallyo/tallyo:latest`).

## Extending (new region or project)

Copy a `live/` directory and edit its `region.hcl` / `project.hcl`. Zero
`infra/modules/` changes.

## Cost warning

`plan`/`apply` against real cloud require credentials, an existing project with
active billing, and the bootstrap state bucket. They create billable resources
(Cloud SQL instance, Cloud Run, Artifact Registry, Secret Manager). Only the
credential-free verification above is safe to run unattended.
